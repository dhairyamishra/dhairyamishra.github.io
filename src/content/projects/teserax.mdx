## Overview

Teserax is a **solo-built agentic reasoning product** that I designed, implemented, and shipped end-to-end. The product turns linear LLM chat into a persistent visual graph that users can branch, merge, prune, cross-link, summarize, and rewrite. Instead of building another chat wrapper, I defined the core interaction model, the graph behavior, the AI orchestration layer, the persistence model, and the deployment architecture myself.

What makes Teserax a strong representation of my work is not just that it works — it is that I owned the full problem alone: product definition, UX model, graph architecture, streaming backend, multi-model support, BYOK, persistence, auth, deployment, testing, and operational guardrails. The result is a live system that demonstrates both **product ownership** and **deep technical range** across frontend, backend, AI orchestration, infrastructure, and quality engineering.

## What I Owned

- **Product definition**: Chose to solve non-linear reasoning and analysis work as a graph-native AI product rather than a conventional chat interface
- **Core interaction model**: Designed the Global Panel as the single control surface for graph creation and mutation
- **Graph system design**: Built the node model, structural/cross-link edge system, deterministic layout behavior, and reversible graph operations
- **AI orchestration**: Implemented the planner/executor pipeline that classifies natural-language intent, previews edits, and executes graph mutations
- **Platform architecture**: Designed the split deployment across Vercel, Railway, and Supabase with same-origin API routing and local-only fallback mode
- **Quality and operations**: Added auth, validation, rate limits, metrics, observability, and automated test coverage across critical user flows

## Hard Problems Solved

- **Inventing a graph-native AI interaction model**: The product had to feel more powerful than chat without becoming harder to use. I solved that by centralizing control in one Global Panel while projecting results into a visual graph.
- **Executing graph edits safely**: Natural-language instructions can create, mutate, merge, or remove graph structures. I built a DAG-aware planner/executor that previews non-trivial edits, respects dependencies, and keeps the full action reversible through a single undo transaction.
- **Keeping conversation and graph state synchronized**: Teserax uses a dual-lane model linking chat messages to graph mutations through `projectedNodeIds`, so users can move between conversational intent and graph artifact without losing context.
- **Supporting real deployment constraints**: The system needed streaming responses, longer-lived requests, auth, persistence, and reliable routing. I split the architecture across Vercel, Railway, and Supabase so each layer matched the operational constraints of the workload.
- **Balancing flexibility with reliability**: I added multi-model support, BYOK, optional Brave Search augmentation, structured validation, rate limiting, security headers, and observability so the product is not just interesting, but operable.

## Key Features

- **Global Panel + Visual Graph**: One chat-driven control surface reshapes a persistent graph of typed nodes, with every message linked to the nodes it created or changed
- **DAG-Aware Planner/Executor**: Teserax classifies instructions, previews multi-step graph edits, and executes dependency-aware plans in parallel where possible, all wrapped in a single undo transaction
- **Graph Operations**: Supports branching, merging, summarization, cross-linking, pruning, and in-place node rewriting from natural-language commands
- **Deterministic Canvas System**: React Flow canvas with swim lanes, collision-aware tree layout, structural DAG edges, and non-structural relationship links
- **Multi-Model + BYOK**: OpenAI and xAI/Grok through one OpenAI-compatible adapter, plus bring-your-own-key support for additional providers with encrypted key storage
- **Brave Search Auto-RAG**: Optional web augmentation distills each prompt into parallel Brave Search queries and injects cited live context into the model call
- **Media-Aware Reasoning**: Auto-spawns media satellite nodes for images, tables, video, charts, code, data, and PDFs; supports PDF ingestion for "chat with PDF" workflows
- **Cloud and Local Persistence**: Supabase-backed auth, storage, sharing, and history when configured, with a full local-only mode when cloud services are absent
- **Import / Export + Templates**: JSON, PNG, Markdown export; imports from Teserax JSON, ChatGPT exports, Markdown, and plain text, plus built-in analysis templates
- **Operational UX**: Undo/redo, command palette, onboarding, dark/light theme, mobile bottom-sheet panel, and a Main Thread workspace for assembling final outputs

## Architecture

Teserax uses a **dual-lane response model**. Every LLM operation emits immutable chat messages in parallel with graph nodes, linked via `projectedNodeIds`. The graph stores both raw provider responses and derived structured projections, so users can move between the conversational history and the graph artifact without losing context.

**Core loop**: user sends an instruction in the Global Panel → the planner classifies intent and proposes a graph action plan → Teserax previews non-trivial edits → the execution pipeline runs graph mutations with SSE-backed model responses → changed nodes are focused on the canvas and the whole action remains reversible through one undo.

The deployed architecture is split intentionally: **Vercel** serves the React/Vite frontend, **Railway** runs the persistent Hono backend for SSE streaming and longer-lived LLM requests, and **Supabase** provides Postgres, auth, and row-level security. The frontend calls relative `/api/*`, which Vercel rewrites to Railway so the browser still sees a same-origin API. That architecture choice reflects product judgment as much as implementation detail: each layer is chosen to fit the actual workload rather than forcing everything into one generic hosting model.

## Infrastructure & Quality

- **API Server**: Hono on Node, with structured output validation, ES256/HS256 JWT auth, rate limiting, budget controls, security headers, audit logging, and health/metrics endpoints
- **Hosting**: Frontend on Vercel, backend on Railway, and data/auth on Supabase; local development runs as Vite + Hono with proxying between the two processes
- **Testing**: Vitest for unit/integration coverage plus Playwright for core journeys, auth, BYOK, canvas behavior, export/import, keyboard shortcuts, responsive flows, and mobile use cases
- **CI/CD**: GitHub Actions quality gates, frontend build validation, and deployment wiring across managed hosting services
- **Observability**: In-memory execution tracing, metrics APIs, Sentry on frontend and backend, and optional Plausible analytics
